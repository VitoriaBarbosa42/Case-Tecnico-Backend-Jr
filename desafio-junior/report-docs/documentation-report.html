📘 Documentação Técnica do Projeto

Guia Completo de Instalação, Arquitetura e Decisões de Desenvolvimento

Visão Geral

O projeto consiste em uma aplicação backend desenvolvida com Java 21 e Spring Boot 3.3.4. Ela atua como um BFF (Backend for Frontend) de integração com a API pública do ViaCEP.

Seu principal objetivo de negócio é servir como um canal de consulta rápida de endereços para gerentes de consignado na rede de agências, permitindo validar de forma consistente se o endereço fornecido pelo cliente condiz com seu histórico no banco de dados durante a venda de produtos.

Funcionalidades Principais

  • READ Leitura estruturada de informações de logradouro via CEP.
  • Sanitização Remoção de campos desnecessários ao caso de uso (ibge, gia, ddd e siafe).
  • Normalização Conversão do campo logradouro para lowercase (letras minúsculas).
  • Robustez Tratamento de erros centralizado com respostas em JSON estruturado para CEPs inválidos e inexistentes.

Refinamento Técnico

As especificações de negócio foram baseadas nas regras de refinamento técnico acordadas:

Requisito Detalhe da Implementação
Integração com API Externa Consumo do endpoint https://viacep.com.br/ws/{cep}/json/ usando Spring Cloud OpenFeign.
Exclusão de Campos Os campos ibge, gia, ddd e siafe são omitidos da resposta retornada ao cliente.
Modificação de Dados O campo logradouro deve ser retornado em minúsculas (lowercase).
Validação do CEP A entrada é higienizada (remoção de hífens) e validada em relação a hífens e tamanho exato de 8 dígitos numéricos.

Contexto do Gerente de Consignado

O gerente consultivo utiliza essa API durante a venda do produto para checar e registrar se o endereço informado pelo cliente bate com as bases cadastrais históricas do banco, mitigando fraudes e inconsistências de correspondência.

🚀 Como Baixar e Executar Localmente

  1. Clonar o Repositório

    Faça o download do código-fonte para o seu ambiente local:

    git clone https://github.com/VitoriaBarbosa42/Case-Tecnico-Backend-Jr.git cd Case-Tecnico-Backend-Jr/desafio-junior

  2. Compilar e Testar

    Compile a aplicação e execute os testes automatizados com o Maven Wrapper:

    ./mvnw clean test

  3. Executar a Aplicação

    Suba o servidor Spring Boot local na porta padrão 8080:

    ./mvnw spring-boot:run

    No Windows, utilize mvnw.cmd em vez de ./mvnw.

  4. Testar Endpoints no Navegador

    Acesse o endpoint de busca de CEP localmente em seu navegador:

    http://localhost:8080/viacep/01001-000

🏛️ Arquitetura & Decisões ("O Porquê")

O sistema foi estruturado seguindo as boas práticas da arquitetura **BFF (Backend for Frontend)** e **MVC (Model-View-Controller)**.

1. Spring Cloud OpenFeign

Em vez de configurar RestTemplate ou WebClient de forma manual, optou-se pelo OpenFeign por ser declarativo, de simples leitura e facilitar o acoplamento dinâmico de configurações de timeouts e fallbacks, além de reduzir drasticamente o código boilerplate.

2. Java Records (DTOs)

Utilizados para modelar o DTO de entrada/saída (ViaCepResponseDTO e ExceptionsDTO). Garante a imutabilidade por design, gerando automaticamente getters, equals, hashCode e toString, além de permitir o processamento automático da string de logradouro para lowercase de forma transparente.

3. Validação Centralizada e Tratamento Global

As validações de input são definidas no Controller via Bean Validation (@NotBlank, @Pattern) e complementadas de forma robusta na camada de Service para garantir o isolamento das regras. O GlobalExceptionHandler intercepta as falhas para retornar respostas consistentes e padronizadas em formato JSON (contendo timestamp, status, erro amigável, mensagem e caminho da requisição).
