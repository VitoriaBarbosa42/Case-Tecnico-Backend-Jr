⚡ Achados Críticos

1. Acoplamento de Validação de Negócio e Lançamento de Exceções no Construtor do Record DTO
Crítico Model / DTO
ViaCepResponseDTO.java (Linhas 12-18)
Problema

O construtor canônico do record lança IllegalArgumentException("CEP não encontrado") se o logradouro for nulo ou vazio. Isso gera dois problemas graves:
1. Viola a semântica de Records, que devem ser meros portadores de dados (data carriers) sem efeitos colaterais.
2. Impede a representação de CEPs válidos de municípios menores onde o logradouro é vazio de forma legítima. Além disso, quebra os testes unitários da classe ViaCepResponseDTOTest que verificavam o comportamento de manter nulo/vazio.

2. Divergência Completa entre Serviço e Testes Unitários (Validações Ausentes)
Crítico Service / Tests
ViaCepService.java (Linhas 16-19) e ViaCepServiceTest.java
Problema

A classe ViaCepService não realiza qualquer validação de entrada de CEP (nulo, vazio, ou formato inválido) nem tratamento de erros para CEP não encontrado. Entretanto, seus testes unitários correspondentes (ViaCepServiceTest) testam exaustivamente se esses cenários lançam IllegalArgumentException. Com isso, os testes do serviço falham ao serem executados na build, impossibilitando uma integração contínua (CI) verde.

3. Falha de Execução de Mockito em Ambientes de JDK Modernos (JDK 21/24)
Crítico Tests / Infra
Pom.xml & Configuração do Surefire
Problema

A execução dos testes unitários utilizando Mockito falha por padrão no JDK 24 (e gera avisos sérios no JDK 21) devido a restrições de carregamento dinâmico de agentes nativos da JVM. O Mockito tenta instanciar inline mocks e é bloqueado. Para contornar e estabilizar os testes, precisamos declarar explicitamente o uso do subclass mockmaker nas configurações de teste ou passar parâmetros na JVM.

Solução Sugerida

Criar o arquivo de configuração src/test/resources/mockito-extensions/org.mockito.plugins.MockMaker contendo mock-maker-subclass. Isso desativa a tentativa de inline mocking que gera o erro do dynamic agent loading.

Impacto Técnico Falha geral na build do Maven que impede a validação automatizada.
Atributos Afetados Portabilidade, Confiabilidade da Build
Esforço de Correção Baixo (Arquivo de configuração de Mockito criado com sucesso)

1. Testes de Controller Frágeis (Ausência de MockMvc)
Importante Controller / Tests
ViaCepControllerTest.java
Problema

O teste atual do controlador (shouldReturnAddressWhenCepIsValid) instancia o controlador diretamente como um objeto Java puro. Com isso, não são validadas as regras HTTP do Spring MVC, as rotas do endpoint (/viacep/{cep}), nem as validações do Bean Validation aplicadas via anotações como @NotBlank e @Pattern no parâmetro do endpoint.

Solução Sugerida

Refatorar a classe ViaCepControllerTest para utilizar @WebMvcTest(ViaCepController.class) injetando MockMvc. Isso permite testar as validações de input, as respostas de código de status HTTP reais e o mapeamento adequado de rotas.

Impacto Técnico Validadores de CEP em anotações no Controller podem ser bypassados sem testes de regressão correspondentes.

2. Tratamento Inadequado de Exceções de APIs Parceiras (FeignException)
Importante Error Handler
GlobalExceptionHandler.java
Problema

O GlobalExceptionHandler captura NullPointerException e o traduz para um Bad Request. Capturar NPE é uma má prática de Clean Code que pode camuflar falhas reais da aplicação. Além disso, não há tratamento específico para exceções lançadas pelo Feign Client (feign.FeignException). Caso o serviço ViaCEP esteja inoperante ou retorne um erro HTTP não tratado, a aplicação vai expor o erro bruto em 500 sem uma formatação limpa e amigável para o cliente do BFF.

Solução Sugerida

1. Remover a captura de NullPointerException e focar no tratamento adequado do fluxo de dados.
2. Implementar um método tratador de exceção para FeignException, formatando a resposta de erro com o DTO padronizado e indicando o status correspondente (ex: 502 Bad Gateway para falhas na API parceira).

Impacto Técnico Vazamento de stack traces e respostas não padronizadas no BFF em caso de falha da API integrada.
Atributos Afetados Confiabilidade, Resiliência, Robustez
Esforço de Correção Médio

💡 Sugestões de Melhoria

1. Configuração de Timeouts e Resiliência no Feign Client
Sugestão BFF / Client
ViaCepClient.java
Descrição

A interface do Feign Client não possui nenhuma configuração de timeout para conexão (Connect Timeout) ou para leitura dos dados (Read Timeout). Por padrão, dependendo das configurações do Feign, a aplicação pode aguardar tempo indeterminado caso a API do ViaCEP trave ou tenha lentidão extrema.

Solução Recomendada

Definir os timeouts padrão do Feign no arquivo application.properties: 

feign.client.config.default.connectTimeout=3000
feign.client.config.default.readTimeout=3000

Impacto Técnico Aumenta a estabilidade geral da aplicação sob falhas do sistema integrado.
Atributos Afetados Performance, Resiliência
Esforço de Correção Muito Baixo

➡️ Plano de Ação & Roadmap de Melhorias

  • 1

    Ajustar Construtor do DTO (ViaCepResponseDTO)

    Remover o lançamento da exceção do construtor canônico. Evitar que o record tenha lógica de negócio e permitir logradouros nulos/vazios sem crashes.

  • 2

    Implementar Validações no Serviço (ViaCepService)

    Adicionar validação de nulo, vazio e formato do CEP antes da chamada do Feign. Implementar a checagem da flag erro vinda da resposta do ViaCEP.

  • 3

    Aprimorar o Tratador Global de Exceções

    Remover o tratamento genérico de NullPointerException e adicionar interceptores para anotações de validação de rota e exceções de chamadas de rede externas (FeignException).

  • 4

    Refatorar Testes do Controller para Usar MockMvc

    Substituir o teste manual do controller por testes web que usem MockMvc, testando de ponta a ponta as validações de input nos endpoints de entrada do BFF.
